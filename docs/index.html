02 — Why SQL Forge

Built for the BigQuery + dbt ecosystem

Generic text-to-SQL models are trained on Spider benchmark data. SQL Forge is purpose-built for real-world data-warehouse SQL.

🔧
Generic SQLCoder / Defog
Spider · WikiSQL trained
❌
No BigQuery dialect — uses ANSI SQL; generates DATE_FORMAT() instead of FORMAT_DATE()
❌
No QUALIFY — rewrites deduplication as verbose nested subqueries instead of QUALIFY ROW_NUMBER()=1
❌
No dbt macros — cannot generate {{ ref() }}, {% if is_incremental() %}, or {{ config() }}
❌
No UNNEST syntax — BigQuery array-unnesting pattern is absent from training data
✅
General SQL coverage — good at generic SELECT / JOIN / GROUP BY across many SQL dialects
🔥
SQL Forge
BigQuery + dbt trained
✅
Native BigQuery dialect — FORMAT_DATE, DATE_TRUNC, TIMESTAMP_DIFF, EXTRACT, backtick table refs
✅
QUALIFY clause — uses BigQuery's QUALIFY ROW_NUMBER() OVER (...) = 1 pattern natively
✅
dbt Jinja macros — generates {{ ref() }}, {{ source() }}, {% if is_incremental() %}, {{ config() }}
✅
UNNEST + ARRAY_AGG — correct BigQuery array expansion with proper UNNEST(col) AS alias syntax
✅
DPO anti-pattern training — explicitly trained away from 6 common BigQuery mistakes via preference optimization

-- Question: "Return only each customer's most recent order"
-- SQL Forge generates:

SELECT
  order_id,
  customer_id,
  created_at,
  amount_usd,
  status
FROM `project.ecommerce.orders`
WHERE status NOT IN ('cancelled', 'refunded')
QUALIFY
  ROW_NUMBER() OVER (
    PARTITION BY customer_id
    ORDER BY created_at DESC
  ) = 1

-- SQL Forge generates:

{{ config(
  materialized = 'incremental',
  unique_key   = 'date_day',
  tags         = ['revenue', 'daily']

WITH orders AS (
  SELECT
    DATE_TRUNC(created_at, DAY) AS date_day,
    SUM(amount_usd)             AS revenue_usd,
    COUNT(DISTINCT order_id)    AS order_count
  FROM {{ ref('stg_orders') }}
  WHERE status != 'cancelled'
  {% if is_incremental() %}
    AND created_at > (
      SELECT MAX(date_day) FROM {{ this }}
    )
  {% endif %}
  GROUP BY 1
)
SELECT * FROM orders
-- Question: "Sum revenue per product from the items array in orders"
-- SQL Forge generates:

SELECT
  item.sku                        AS product_sku,
  SUM(item.qty * item.unit_price) AS total_revenue_usd,
  COUNT(DISTINCT o.order_id)      AS order_count
FROM `project.ecommerce.orders` o
  , UNNEST(o.items) AS item        -- ← correct BigQuery array unnesting
WHERE o.status = 'delivered'
  AND o.created_at >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
GROUP BY 1
ORDER BY total_revenue_usd DESC
LIMIT 20

-- Generic model often generates (wrong):
-- FROM orders, orders.items AS item  ← invalid in BigQuery
-- Question: "Show monthly retention cohorts for the past year"
-- SQL Forge generates:

WITH cohorts AS (
  SELECT
    u.user_id,
    DATE_TRUNC(u.created_at, MONTH)     AS cohort_month,
    DATE_TRUNC(e.event_time, MONTH)     AS activity_month
  FROM `project.ecommerce.users` u
  JOIN `project.ecommerce.events` e USING (user_id)
  WHERE e.event_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 365 DAY)
)
SELECT
  FORMAT_DATE('%Y-%m', cohort_month)                        AS cohort,
  DATE_DIFF(activity_month, cohort_month, MONTH)            AS months_since_signup,
  COUNT(DISTINCT user_id)                                   AS retained_users,
  COUNT(DISTINCT user_id) / FIRST_VALUE(COUNT(DISTINCT user_id))
    OVER (PARTITION BY cohort_month
          ORDER BY DATE_DIFF(activity_month, cohort_month, MONTH)
          ROWS UNBOUNDED PRECEDING)                         AS retention_rate
FROM cohorts
GROUP BY 1, 2
QUALIFY DATE_DIFF(activity_month, cohort_month, MONTH) BETWEEN 0 AND 11
ORDER BY cohort, months_since_signup
